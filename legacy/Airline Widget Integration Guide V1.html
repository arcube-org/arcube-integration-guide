Airline Widget Integration Guide

Version 1.0.0

Summary

This guide explains how to add the Arcube booking widget to your airline platform. It shows where to place the widget in the booking flow and what parts of the flow remain under your control. It outlines the system architecture and lists each step in the integration. It gives examples of cart events for different ancillaries and includes new examples for lounge and insurance items. It also provides guidance on expected messages from the widget.

Overview

This guide explains how to integrate the Arcube booking widget into an airline booking platform. The widget is an embedded ancillary experience that appears inside the airline booking flow. It helps passengers discover and select additional services such as airport transfers, car rental, lounge access, eSIMs, carbon offset, insurance and other ancillary products.

The widget does not replace the airline booking engine. The airline still owns:

·         the booking session

·         the flight selection

·         the basket

·         the checkout

·         the final order

The widget adds a passenger‑facing ancillary layer on top of the airline booking journey.

In most integrations, the widget is shown on the extras or ancillary step. A common pattern includes:

·         Loading the Arcube integration script on the booking site

·         Waiting until the correct booking step and booking data are available

·         Sending the booking context to Arcube

·         Opening the widget inside the airline page

·         Fetching offers and showing them to the passenger

·         Letting the passenger select services

·         Handing selected items back to the airline basket flow

Operating model:

·         Prepare recommendations as soon as there is a stable booking session

·         Render the widget on the extras step where the passenger is expected to buy ancillaries

Authentication and Token Flow

Token Flow

This section describes the preferred widget authentication model for airline integrations.

 

Figure 1: Secure token vending flow via airline CDN proxy

The recommended approach keeps the airline secret on the airline side, and the browser receives only a short-lived token. the signed flow below is the preferred tenant-facing model. In the current model, the widget token is valid for 4 hours.

 

When the booking page needs the widget, the browser first asks an airline-controlled endpoint for a widget token. That request stays inside the airline environment. The airline secure layer, such as a backend service, CDN worker, or edge layer, builds a canonical string and signs it with HMAC using the airline secret. Arcube receives that signed request, verifies it, and returns a short-lived token. The browser then passes the token to the widget, and the widget uses it for later Arcube API calls.

 

The reference image below shows the same flow in a more visual way. The important point is that the airline website does not expose the secret in browser code. The secure layer signs the request, forwards it to Arcube, and returns only a token to the browser.

 

 

Figure 2: Token flow sequence diagram

The signed text usually contains the tenant identifier, a timestamp, and a session ID or PNR when one already exists. Arcube rebuilds the same string on its side and checks that the signature matches. It also checks that the tenant is known and that the timestamp or nonce is still valid.

 

The airline secret and the signing logic stay on the airline secure side. The browser must never receive them. The browser should receive only the short-lived widget token and the booking context it already owns.

 

If a session ID or PNR exists before the widget opens, it should be included in the signed request so the token can be tied to that booking context. If a PNR does not exist yet, a session-level identifier is enough for the first step.

 

For this token, the safest approach is to keep it in memory and treat it as temporary. In the current model, the token is valid for 4 hours. Where possible, it should also be tied to the session or PNR.

System Architecture

The tenant integration has four moving parts:

·         the airline frontend page

·         the Arcube integration script

·         the Arcube widget

·         the airline basket API

The airline page mounts the widget. The widget sends cart events back to the page, and the airline page updates its own basket.

Figure 3: Architecture diagram

Integration Flow

1. User opens the widget

The airline renders the widget component when the user reaches the extras step. Mount the widget when the extras route is active and destroy it when that view unmounts.

2. User searches for services

The widget uses the session and flight context from the airline page. For some ancillaries, the passenger also enters extra product‑specific details. Examples include pickup and drop‑off addresses or driver age.

3. User selects an option

The widget keeps the passenger inside the airline page and lets them select the ancillary product.

4. Add to cart

When the passenger adds an item, the widget emits a cart event to the parent page. The airline frontend receives that event and calls its own basket API.

5. Continue to checkout

After the airline basket is updated successfully, the normal airline flow continues.

Add to Cart Functionality

Styling and Branding:

The widget can be branded to match the airline experience. The current styling contract is based on a hosted theme file, not a full custom stylesheet for the whole widget. The airline hosts a theme JSON file on an HTTPS URL and passes that URL to the widget as themeJsonUrl when the widget is opened.

The widget reads that theme file, applies the color and typography tokens internally, and keeps the styling scoped to the widget itself. If the airline wants to use a custom font, the theme file can also include a font-url value. This is an HTTPS stylesheet link, such as a hosted font stylesheet or a Google Fonts URL. The widget loads that stylesheet automatically and uses the declared font-family.

In practice, the theme file should include an en block indicating the language with a colors section and a typography section. At minimum, the widget expects primary, secondary, bodyBg, neutral, black, white, link, and typography.font-family. Optional color values such as success, warning, danger, info, and lightnessThreshold may also be provided.

When the airline opens the widget, it should pass a themeJsonUrl value that points to its hosted theme file. In a typical integration, that value looks like this:

themeJsonUrl=https://cdn.airline.com/arcube/widget-theme.json

In the current implementation, the query-based theme URL supports the main color tokens such as primary, secondary, bodyBg, neutral, black, white, link, success, warning, danger, info, and lightnessThreshold, as well as fontFamily and fontUrl. This makes it useful for quick branding changes. For long-term production control, a hosted theme JSON file is still the cleaner option.

 

Example theme file:

If the theme file cannot be fetched, is invalid, or is missing required values, the widget falls back to the default Arcube styling. For that reason, the theme file should be hosted on a stable HTTPS URL and treated as part of the production integration.

Frontend Integration Lifecycle Rules

·         Load the widget only on the route where it is needed

·         Destroy the widget when the route unmounts

·         Re‑initialize the widget when the booking session changes

·         Keep the mount container stable

Error Handling

The widget must never block the airline booking flow. Follow these rules to handle errors:

·         If the widget is not available, hide the widget area

·         If the widget emits an “arcube:widget:error” event, log it and keep the airline flow usable

·         If your cart update fails, show a clear message and do not assume the ancillary was saved

·         If session data is missing, do not mount the widget
