Below are the essential API calls which allow interaction with Arcube's AI Recommendation Engine. For each integration method, we indicate which API calls are required.

## Bearer Token

All calls require a valid **Bearer** token obtained from the authentication endpoint. The required credentials are **client_id** and **client_secret**, which will be provided by Arcube. See: [Authentication](/docs/reference/authentication).

## 6.1 Ingest Passenger Data

### Description

Insert trip and passenger context into Arcube to enable personalization.

<Callout type="info" title="Request URL Parameter">
  Replace `${arcube-url}` with your assigned Arcube environment URL (e.g., `https://api.arcube.com`).
</Callout>

<Callout type="info" title="HTTP Method: POST">
  **URL:** `${arcube-url}/aggregation-api/servePassengerRecommendations`
</Callout>

<Callout title="Request Headers">
  - `Authorization: Bearer <access_token>`
  - `Content-Type: application/json`
  - `Idempotency-Key: <uuid>` *(recommended)*
</Callout>

<Callout type="info" title="HTTP Response">
  `200 OK`
</Callout>

<Callout type="warn" title="One endpoint, two behaviors">
  Ingesting passenger data and serving recommendations are the **same endpoint** (`servePassengerRecommendations`) - `pointOfSale` selects the behavior. When it is `API`, Arcube treats the call as **PNR ingestion / generation only**: the trip and passenger context is stored and **no recommendations are returned** (retrieve them later via [6.2 Serve Recommendations](#62-serve-recommendations)). For any other value (`AIRLINE_WEBSITE`, `EMAIL`, `WIDGET`, `MOBILE_APP`, ...) Arcube ingests the context **and** returns recommendations in the same response.
</Callout>

### Notes

- `pointOfSale` is **required**. See the callout above for how it changes the call's behavior.
- **Only `id` and `type` are essential per passenger.** All other passenger fields - `firstName`, `lastName`, `email`, `accompaniedInfant`, `associatedAdultId`, `isLead` - are optional. Send them only if you want richer personalization or transactional/marketing emails. This removes any sensitive-data restriction on the call.
- **`passengers` and `segments` are required only on the first ingest for a PNR.** Subsequent calls for the same PNR may omit them, or send only the fields that changed.
- Include the **lead passenger's email** (`isLead: true` + `email`) if you want Arcube to send transactional/marketing emails.
- `tenant` is **optional** - your tenant is resolved from the Bearer token. Send it only when you need to override.
- `fare`, `pnrLocale`, and `pnrCurrency` are **optional** enrichment fields: `fare` is the cabin/fare class (e.g. `business`), `pnrLocale` is the locale used for generated content and emails (e.g. `es-es`), and `pnrCurrency` is the ISO 4217 currency used for pricing (e.g. `USD`).

## 6.2 Serve Recommendations

### Description

Retrieve recommendations for an already-ingested PNR. This is the **same endpoint** as [6.1 Ingest Passenger Data](#61-ingest-passenger-data), called with a non-`API` `pointOfSale`.

<Callout type="info" title="Request URL Parameter">
  Replace `${arcube-url}` with your assigned Arcube environment URL (e.g., `https://api.arcube.com`).
</Callout>

<Callout type="info" title="HTTP Method: POST">
  **URL:** `${arcube-url}/aggregation-api/servePassengerRecommendations`
</Callout>

<Callout title="Request Headers">
  - `Authorization: Bearer <access_token>`
  - `Content-Type: application/json`
</Callout>

<Callout type="info" title="HTTP Response">
  `200 OK`
</Callout>

### Request Payload Sample

```json
{
  "pnr": "3MIN12",
  "pointOfSale": "WIDGET"
}
```

`pointOfSale` identifies where the offers will be surfaced (e.g. `AIRLINE_WEBSITE`, `EMAIL`, `WIDGET`, `MOBILE_APP`); new values may be introduced without breaking the contract. Setting it to `API` instead ingests without returning offers (see [6.1](#61-ingest-passenger-data)). Your tenant is resolved from the Bearer token, so it is not part of the payload.

### Request Payload JSON Schema

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",

  "required": ["pnr", "pointOfSale"],
  "properties": {
    "pnr": { "type": "string" },
    "pointOfSale": { "type": "string" }
  }
}
```

## 6.3 Verify Prices

### Description

Call this endpoint **immediately before order creation** to confirm that the prices held in the customer's cart still match Arcube's authoritative bundle. Prices may drift between the original quote and checkout - a vendor changes their rate, a supplier updates pricing, or an FX/markup/eligibility policy is updated.

The response indicates whether each line item is still valid and returns the authoritative price for any item that has drifted, so the client can re-confirm with the user before charging.

<Callout type="info" title="Request URL Parameter">
  Replace `${arcube-url}` with your assigned Arcube environment URL (e.g., `https://api.arcube.com`).
</Callout>

<Callout type="info" title="HTTP Method: POST">
  **URL:** `${arcube-url}/checkout/verify`
</Callout>

<Callout title="Request Headers">
  - `Authorization: Bearer <access_token>`
  - `Content-Type: application/json`
</Callout>

<Callout type="info" title="HTTP Response">
  `200 OK` - verification ran. Inspect `verified` and `mismatches` in the body to determine whether the cart is safe to charge.
</Callout>

<Callout type="warn" title="Allowed roles">
  The Bearer token must belong to one of: `api_client`, `widget_user`, `whitelabel_user`.
</Callout>

### Notes

- `correlationId` references the pricing bundle returned by the earlier batch-price call. UUID-shaped by default; any non-empty string is accepted.
- All items in a single request must share the same `currency` (3-char ISO 4217).
- `extra[]` carries add-ons attached to a parent item (e.g. lounge guest passes). Extras inherit the parent's `categoryCode`.
- A `200` response with `verified: false` is **not an error** - it means the cart needs to be re-priced on the client. `items[]` carries the authoritative prices to re-display; `mismatches[]` explains what drifted.
- Recommended `categoryCode` values: `ESIMS`, `LOUNGE_ACCESS`, `FAST_TRACK`, `BAGGAGE_PROTECTION`, `CARBON_OFFSET`, `AIRPORT_TRANSFER`, `CAR_RENTAL`.
- `quantity` is capped at 100 per line item.


### Response Field Reference

**Top level**

| Field           | Type    | Notes                                                        |
|-----------------|---------|--------------------------------------------------------------|
| `verified`      | boolean | `true` when every line item matches the authoritative bundle |
| `correlationId` | string  | Echoes the request `correlationId`                           |
| `items`         | array   | Authoritative item prices; lookup-failed items are omitted   |
| `mismatches`    | array   | Per-item drift details; empty when `verified === true`       |

**`items[]` entry**

| Field          | Type             | Notes                                                       |
|----------------|------------------|-------------------------------------------------------------|
| `id`           | string           | Item id from the request                                    |
| `categoryCode` | string           | Authoritative category from the bundle                      |
| `unitPrice`    | number           | Authoritative unit price                                    |
| `quantity`     | integer          | Echoes request                                              |
| `totalPrice`   | number           | Authoritative total = `unitPrice × quantity` (rounded to 2) |
| `currency`     | string (3 chars) | Request currency                                            |
| `extra`        | array            | Optional; same shape minus `categoryCode`                   |

**`mismatches[]` entry**

| Field       | Type                                                              | Notes                                                              |
|-------------|-------------------------------------------------------------------|--------------------------------------------------------------------|
| `itemId`    | string                                                            | Always present                                                     |
| `extraId`   | string \| null                                                    | `null` when the mismatch is on the main item; set when on an extra |
| `field`     | `"unitPrice"` \| `"totalPrice"` \| `"categoryCode"` \| `"lookup"` | Which field drifted                                                |
| `submitted` | number \| string \| null                                          | Client-supplied value (`null` for `lookup` failures)               |
| `expected`  | number \| string \| null                                          | Authoritative value (`null` for `lookup` failures)                 |
| `reason`    | `"VALUE_MISMATCH"` \| `"NOT_IN_BUNDLE"`                           | `NOT_IN_BUNDLE` accompanies `field: "lookup"`                      |

### Error Responses

Validation, auth, and upstream errors use the standard backend envelope (see [Error Model](/docs/reference/error-model)):

```json
{
  "success": false,
  "error": "correlationId is required; items.0.unitPrice must be greater than or equal to 0",
  "statusCode": 400,
  "traceId": "abc123",
  "correlationId": "req-7f1e",
  "timestamp": "2026-05-18T10:23:45.123Z"
}
```

Business-level pricing errors are passed through unchanged with a `code` discriminator:

```json
{
  "statusCode": 404,
  "message": "Quote bundle not found or expired.",
  "code": "QUOTE_EXPIRED"
}
```

| Status | Origin          | Trigger                                                                  |
|--------|-----------------|--------------------------------------------------------------------------|
| 200    | pricing-service | Verification ran - inspect `verified` and `mismatches`                   |
| 400    | arcube-backend  | Zod validation failure (malformed body)                                  |
| 400    | pricing-service | `MIXED_CURRENCY`, `UNSUPPORTED_CURRENCY`, `BUNDLE_INTEGRITY_ERROR`       |
| 401    | arcube-backend  | Missing or invalid token                                                 |
| 403    | pricing-service | `AIRLINE_MISMATCH` - bundle does not belong to the authenticated airline |
| 404    | pricing-service | `QUOTE_EXPIRED` - bundle not found or expired                            |
| 5xx    | arcube-backend  | Upstream verification service unreachable / timeout                      |

Business error codes: `QUOTE_EXPIRED`, `AIRLINE_MISMATCH`, `MIXED_CURRENCY`, `UNSUPPORTED_CURRENCY`, `BUNDLE_INTEGRITY_ERROR`.

<Callout type="warn" title="Client behavior on drift">
  When `verified` is `false`, **do not proceed to order creation**. Re-render the cart using `items[]` from the response, ask the user to re-confirm the new totals, then call verify again before charging.
</Callout>

## 6.4 Confirm Order

### Description

Confirm an order after **successful payment**. Arcube will place orders with relevant vendors and record the transaction.

<Callout type="info" title="Request URL Parameter">
  Replace `${arcube-url}` with your assigned Arcube environment URL (e.g., `https://api.arcube.com`).
</Callout>

<Callout type="info" title="HTTP Method: POST">
  **URL:** `${arcube-url}/order-api/confirmOrder`
</Callout>

<Callout title="Request Headers">
  - `Authorization: Bearer <access_token>`
  - `Content-Type: application/json`
  - `Idempotency-Key: <uuid>` *(required for safe retries)*
</Callout>

<Callout type="info" title="HTTP Response">
  `200 OK` (full success) - or `207 Multi-Status` / `200` with `status="PARTIAL-SUCCESS"` for partial fulfillment.
</Callout>
