## Authentication and Token Flow

### Token Flow

This section describes the preferred widget authentication model for airline integrations.

<img src="/images/widget/widget-token-flow-diagram.jpg" alt="Secure token vending flow via airline CDN proxy" style={{display: 'block', margin: '0 auto'}} />

<small style={{display: 'block', textAlign: 'center'}}>Figure 1: Secure token vending flow via airline CDN proxy.</small>

The recommended approach keeps the airline secret on the airline side, and the browser receives only a short-lived token. In the current model, the widget token is valid for 4 hours.

When the booking page needs the widget, the browser first asks an airline-controlled endpoint for a widget token. That request stays inside the airline environment. The airline secure layer - such as a backend service, CDN worker, or edge layer - builds a canonical string and signs it with HMAC using the airline secret. Arcube receives that signed request, verifies it, and returns a short-lived token. The browser then passes the token to the widget, and the widget uses it for later Arcube API calls.

The reference image below shows the same flow in a more visual way. The important point is that the airline website does not expose the secret in browser code. The secure layer signs the request, forwards it to Arcube, and returns only a token to the browser.

<img src="/images/widget/widget-token-sequence.jpg" alt="Token flow sequence diagram" style={{display: 'block', margin: '0 auto', width: '100%'}} />

<small style={{display: 'block', textAlign: 'center'}}>Figure 2: Token flow sequence diagram.</small>

The diagram below zooms in on the exact token-generation steps that the Airline CDN/EDGE performs before calling Arcube. The secure layer fetches the airline secret, attaches the required headers (`X-Tenant-Id`, `X-Key-Id`, `X-Timestamp`, `X-Nonce`), prepares the canonical string, signs it, and sends the request with the `X-Arcube-Signature` header. Arcube validates the signature and returns a short-lived token, which the IBE then uses to render the widget.

<img src="/images/widget/widget-token-generation.png" alt="Detailed token generation and HMAC signing steps" style={{display: 'block', margin: '0 auto', width: '100%'}} />

<small style={{display: 'block', textAlign: 'center'}}>Figure 3: Detailed token generation and HMAC signing steps.</small>

The signed text usually contains the tenant identifier, a timestamp, and a session ID or PNR when one already exists. Arcube rebuilds the same string on its side and checks that the signature matches. It also checks that the tenant is known and that the timestamp or nonce is still valid.

The airline secret and the signing logic stay on the airline secure side. The browser must never receive them. The browser should receive only the short-lived widget token and the booking context it already owns.

If a session ID or PNR exists before the widget opens, it should be included in the signed request so the token can be tied to that booking context. If a PNR does not exist yet, a session-level identifier is enough for the first step.

For this token, the safest approach is to keep it in memory and treat it as temporary. In the current model, the token is valid for 4 hours. Where possible, it should also be tied to the session or PNR.

## System Architecture

The tenant integration has four moving parts:

- the airline frontend page
- the Arcube integration script
- the Arcube widget
- the airline basket API

The airline page mounts the widget. The widget sends cart events back to the page, and the airline page updates its own basket.

## Integration Flow

### 1. User Opens the Widget

The airline renders the widget component when the user reaches the extras step. Mount the widget when the extras route is active and destroy it when that view unmounts.

### 2. User Searches for Services

The widget uses the session and flight context from the airline page. For some ancillaries, the passenger also enters extra product-specific details - for example, pickup and drop-off addresses or driver age.

### 3. User Selects an Option

The widget keeps the passenger inside the airline page and lets them select the ancillary product.

### 4. Add to Cart

When the passenger adds an item, the widget emits a cart event to the parent page. The airline frontend receives that event and calls its own basket API.

### 5. Continue to Checkout

After the airline basket is updated successfully, the normal airline flow continues.

## Rendering the Widget

Once the airline has authenticated itself and acquired a short-lived widget token, the airline page can render the widget. The page needs a stable mount element, the Arcube widget bridge script, and a call to `window.Widget.init`.

```html
<div id="arcube-widget"></div>
<script src="PATH_TO_ARCUBE_WIDGET_SCRIPT"></script>
```

Register widget event listeners before initialization when possible, so the parent page can catch early ready, error, and cart events.

```javascript
const widgetConfig = {
  widgetUrl: "ARCUBE_WIDGET_URL",
  tenantId: "YOUR_TENANT_ID",
  sessionId: "YOUR_PNR",
  currency: "USD",
  language: "en-US",
  token: widgetToken,
  themeJsonUrl: "https://cdn.airline.com/arcube/widget-theme.json"
};

window.addEventListener("arcube:widget:ready", (event) => {
  console.log("Arcube widget ready", event.detail);
});

window.addEventListener("arcube:widget:error", (event) => {
  console.error("Arcube widget error", event.detail);
});

window.addEventListener("arcube:cart:updated", (event) => {
  const { cart, correlationId, messageType } = event.detail;
  console.log("Arcube widget cart updated", messageType, correlationId, cart);
});

window.Widget.init(widgetConfig);
```

## Add to Cart Functionality

The widget does not write into the airline basket directly. It emits cart events. The airline frontend must listen to those events and run its own add-to-cart logic. Each widget event contains a cart snapshot with the root `customer` and a flat `selectedAncillaries` array. For add-to-cart, the tenant should catch the widget event, read the newest selected ancillary, and pass that data into its own basket logic.

### Catching the Add Event

Listen for cart update events and extract the latest selected ancillary from the cart. The following example shows how to do that in JavaScript:

```typescript
function getLatestSelectedAncillary(detail: ArcubeCartUpdateEvent) {
  const items = detail.cart.selectedAncillaries;
  return items[items.length - 1] ?? null;
}

window.addEventListener("arcube:cart:updated", (event) => {
  const detail = (event as CustomEvent<ArcubeCartUpdateEvent>).detail;
  if (!detail || detail.messageType !== "CART_ITEM_ADDED") return;

  const latest = getLatestSelectedAncillary(detail);

  if (!latest) return;

  console.log("Widget item added", latest);
  // Pass latest into your own add-to-cart flow.
});
```

## Sending Updates to the Widget

The airline can also send data and commands to the widget. In the current website integration, the airline page already loads the Arcube integration script. That script adds the embedded widget into the page. Runtime communication happens by sending a browser message to the embedded widget window.

```javascript
const widgetUrl = "use the same widget URL configured in your integration";
const widgetOrigin = new URL(widgetUrl).origin;

function getEmbeddedWidgetWindow() {
  const container = document.getElementById("arcube-inline-widget");
  const frame = container?.firstElementChild;
  return frame && "contentWindow" in frame ? frame.contentWindow : null;
}

function sendToWidget(message) {
  const widgetWindow = getEmbeddedWidgetWindow();
  if (!widgetWindow) return;

  widgetWindow.postMessage(
    {
      source: "arcube-parent",
      version: "1.0",
      timestamp: new Date().toISOString(),
      ...message
    },
    widgetOrigin
  );
}

function removeItemFromWidget(itemId) {
  sendToWidget({
    type: "CART_ITEM_REMOVE_REQUEST",
    payload: { itemId }
  });
}

async function handleRemove(itemId) {
  await removeFromAirlineCart(itemId);
  removeItemFromWidget(itemId);
}
```

## Tenant Event Contract

The Arcube widget dispatches events that indicate its state. Tenants must handle these events appropriately. The table below summarizes the events and the required tenant actions:

| Event | Meaning | What the tenant should do |
|-------|---------|--------------------------|
| `arcube:widget:ready` | Widget is ready | Mark the widget area as live |
| `arcube:widget:error` | Widget failed | Show fallback UI and log the error |
| `arcube:cart:updated` + `CART_ITEM_ADDED` | User added an item | Run your add-to-cart logic |
| `arcube:cart:updated` + `CART_ITEM_REMOVED` | User removed an item | Run your remove-from-cart logic |

## Styling and Branding

The widget can be branded to match the airline experience. The current styling contract is based on a hosted theme file. The airline hosts a theme JSON file on an HTTPS URL and passes that URL to the widget as `themeJsonUrl` when the widget is opened.

The widget reads that theme file, applies the color and typography tokens internally, and keeps the styling scoped to the widget itself. If the airline wants to use a custom font, the theme file can also include a `font-url` value - an HTTPS stylesheet link such as a hosted font stylesheet or a Google Fonts URL. The widget loads that stylesheet automatically and uses the declared `font-family`.

The theme file should include an `en` block with a colors section and a typography section. At minimum, the widget expects:

- `primary`, `secondary`, `bodyBg`, `neutral`, `black`, `white`, `link`
- `typography.font-family`

Optional color values - `success`, `warning`, `danger`, and `info` - may also be provided.

When opening the widget, pass a `themeJsonUrl` pointing to the hosted theme file:

```
themeJsonUrl=https://cdn.airline.com/arcube/widget-theme.json
```

The query-based theme URL supports the main color tokens as well as `fontFamily` and `fontUrl`, making it useful for quick branding changes. For long-term production control, a hosted theme JSON file is the cleaner option.

**Example theme file:**

```json
{
  "en": {
    "colors": {
      "primary": "#12ADEF",
      "secondary": "#0A3D62",
      "bodyBg": "#F7FAFC",
      "neutral": "#C8D2DC",
      "black": "#111111",
      "white": "#FFFFFF",
      "link": "#12ADEF",
      "success": "#2E9E5B",
      "warning": "#E8B949",
      "danger": "#D75656",
      "info": "#4AA3C7"
    },
    "typography": {
      "font-family": "Airline Sans, Arial, sans-serif",
      "font-url": "https://cdn.airline.com/fonts/airline-fonts.css"
    }
  }
}
```

If the theme file cannot be fetched, is invalid, or is missing required values, the widget falls back to the default Arcube styling. For that reason, the theme file should be hosted on a stable HTTPS URL and treated as part of the production integration.

## Frontend Integration Lifecycle Rules

- Load the widget only on the route where it is needed
- Destroy the widget when the route unmounts
- Re-initialize the widget when the booking session changes
- Keep the mount container stable

## Error Handling

The widget must never block the airline booking flow. Follow these rules to handle errors:

- If the widget is not available, hide the widget area
- If the widget emits an `arcube:widget:error` event, log it and keep the airline flow usable
- If your cart update fails, show a clear message and do not assume the ancillary was saved
- If session data is missing, do not mount the widget

Listen for error events and wrap your cart update logic in a try-catch block. The following example illustrates how to handle errors in JavaScript:

```typescript
window.addEventListener("arcube:widget:error", (event) => {
  const detail = (event as CustomEvent<{ code: string; message: string }>).detail;
  console.error("Arcube widget error", detail);
  // Hide the widget area or show fallback UI here.
});

async function handleCartAdd(runCartLogic: () => Promise<void>) {
  try {
    await runCartLogic();
  } catch (error) {
    console.error("Cart update failed", error);
    // Show a clear message to the user here.
  }
}
```
