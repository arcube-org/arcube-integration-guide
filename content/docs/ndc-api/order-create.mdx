## Purpose

`OrderCreateRQ` finalizes the selected and priced ancillary purchase.

This request must operate on the result of `OfferPriceRQ`, not directly on raw shopping selections.

- `AirShoppingRQ` discovers offers
- `OfferPriceRQ` confirms and prices them
- `OrderCreateRQ` creates the order using the priced result

## OrderCreateRQ Fields

| Field | Description |
| --- | --- |
| `PointOfSale` | Commercial and sales-channel context of the final order. |
| `Customer.PrimaryContact` | Primary purchaser/contact record. |
| `Passengers[]` | Passenger records associated with the order. |
| `Query.PricedOfferRef` | Reference to the priced offer returned by `OfferPriceRQ`. |
| `Query.OfferItems[]` | The exact priced items to be ordered. |
| `Payments[]` | Payment declaration for the order. |
| `ProductInputs[]` | Any booking-time product inputs still required for fulfillment. |
| `Metadata` | Tenant and session correlation context. |

### Payments

In the example, payment mode is `AIRLINE_CAPTURED`, meaning the airline or seller captures payment externally and passes the payment result into Arcube for order finalization.

### ProductInputs

Examples of booking-time product inputs include:

- digital delivery email for eSIM
- passenger/contact metadata
- provider-required supplemental fields
- flight information for transfer products
- driver or renter data for car rental, where needed

## OrderCreate Processing Model

The `OrderCreateRQ` implementation returns a public `OrderCreateRS` containing:

- `Document`
- `Order`
- `Metadata`

The returned order includes:

- `OrderID`
- `Status`
- `Owner`
- `Items[]`
- `PriceSummary`

The response metadata contains:

- `CorrelationID`
- `PricedSessionID`
- `OrderKey`

This confirms that order creation is driven from a priced-session context and that the public response is intentionally normalized into an NDC-like structure rather than exposing raw provider booking data.
